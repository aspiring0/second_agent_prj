## 1. 文件处理与编码问题

### 问题描述

在上传和处理文档时遇到以下问题：

1. **编码不兼容**：不同来源的 TXT 文件可能使用不同编码（UTF-8、GBK 等），导致读取失败或乱码
2. **Streamlit 文件上传机制与 LangChain Loader 不兼容**：Streamlit 的 `UploadedFile` 对象在内存中，而 LangChain 的 Loader 需要文件路径

### 解决方案

#### 编码容错处理

```python
def _create_text_loader_with_fallback(self, file_path: str) -> TextLoader:
    """创建带有编码容错的 TextLoader"""
    # 优先尝试 UTF-8
    try:
        loader = TextLoader(file_path, encoding="utf-8")
        # 尝试读取一小段来验证编码
        with open(file_path, 'r', encoding='utf-8') as f:
            f.read(1024)
        return loader
    except UnicodeDecodeError:
        # 回退到 GBK（常见于 Windows 中文环境）
        logger.info(f"   📝 检测到非 UTF-8 编码，尝试 GBK...")
        return TextLoader(file_path, encoding="gbk", errors="ignore")
```

#### 临时文件桥接技术

```python
def load_uploaded_files(self, uploaded_files: List[UploadedFile]) -> List:
    """使用临时文件技术适配 LangChain Loader"""
    for up_file in uploaded_files:
        tmp_path = None
        try:
            # 1. 创建临时文件
            suffix = Path(up_file.name).suffix
            with tempfile.NamedTemporaryFile(delete=False, suffix=suffix) as tmp_file:
                tmp_file.write(up_file.getvalue())
                tmp_path = tmp_file.name
            
            # 2. 使用临时文件路径加载
            loader = self._select_loader(tmp_path, up_file.name)
            docs = loader.load()
            
            # 3. 修复 metadata.source 为原始文件名
            for doc in docs:
                doc.metadata["source"] = up_file.name
                
        finally:
            # 4. 清理临时文件
            if tmp_path and os.path.exists(tmp_path):
                os.remove(tmp_path)
```

## 2. 文档切分策略问题

### 问题描述

使用默认的 LangChain 文本切分器处理中文文档时效果不佳：

1. **分隔符不适合中文**：默认分隔符主要针对英文，会在中文句子中间切断
2. **Markdown 文档结构破坏**：代码块、标题等被错误切分
3. **语义断裂**：切分后的片段失去上下文关联

### 解决方案

#### 针对中文优化的分隔符

```python
CHINESE_SEPARATORS = [
    "\n\n",    # 段落分隔（优先级最高）
    "\n",      # 行分隔
    "。",      # 中文句号
    "！",      # 中文感叹号
    "？",      # 中文问号
    "；",      # 中文分号
    "，",      # 中文逗号
    "：",      # 中文冒号
    "(", ")",  # 英文括号
    "（", "）", # 中文括号
    " ",       # 空格
    ""         # 最后按字符切分
]
```

#### 针对 Markdown 优化的分隔符

```python
MARKDOWN_SEPARATORS = [
    "\n\n",    # 段落分隔
    "\n",      # 行分隔
    "```",     # 代码块
    "## ",     # 二级标题
    "### ",    # 三级标题
    "#### ",   # 四级标题
    "- ",      # 列表项
    "* ",      # 列表项

#### 智能切分策略

```python
def split_documents(self, documents: List) -> List:
    """根据文档类型选择不同的分隔符策略"""
    for doc in documents:
        file_type = doc.metadata.get("file_type", "").lower()
        
        # 根据文件类型选择分隔符
        if file_type == ".md":
            separators = MARKDOWN_SEPARATORS
        elif file_type in [".py", ".js", ".java", ...]:
            # 代码文件使用代码分隔符
            separators = ["\n\nclass ", "\n\ndef ", "\n\n", ...]
        else:
            # 普通文档使用中文分隔符
            separators = CHINESE_SEPARATORS
        
        text_splitter = RecursiveCharacterTextSplitter(
            chunk_size=800,
            chunk_overlap=100,  # 重叠防止语义断裂
            separators=separators
        )
```

## 3. 多知识库隔离问题

### 问题描述

系统需要支持多个知识库，但所有文档都存储在同一个向量数据库中：

1. **数据混淆**：不同知识库的文档可能互相干扰
2. **检索错误**：查询知识库 A 时可能返回知识库 B 的内容
3. **删除困难**：删除某个知识库时难以精确删除其所有文档

### 解决方案

#### 元数据过滤实现隔离

```python
# 入库时添加 project_id
def create_vector_db(self, chunks, project_id: str):
    for chunk in chunks:
        chunk.metadata["project_id"] = project_id
    
    vectordb = Chroma(persist_directory=self.persist_dir, ...)
    vectordb.add_documents(documents=chunks)
```

```python
# 检索时按 project_id 过滤
def query(self, question: str, project_id: str, top_k=3):
    results = self.vector_db.similarity_search_with_score(
        question,
        k=top_k,
        filter={"project_id": project_id}  # 元数据过滤
    )
    return results
```

**优势**：
- 单一向量数据库支持多租户
- 数据隔离，互不干扰
- 节省资源（无需为每个知识库创建单独的数据库）

## 4. 多智能体协作问题

### 问题描述

在 Researcher 和 Writer 两个智能体协作过程中遇到：

1. **Writer 无法看到工具返回结果**：Writer 只能看到原始消息，无法获取工具执行的上下文
2. **角色职责混乱**：两个 Agent 的分工不清晰
3. **信息丢失**：研究员查到的信息在传递给作家时丢失

### 解决方案

#### Writer 节点上下文修复

```python
def writer_node(state: AgentState):
    messages = state["messages"]
    
    # 【关键修复】：将历史消息转换为字符串
    # 这样作家才能真正"看见"研究员查到了什么
    conversation_str = "\n".join([
        f"【{msg.type}】: {msg.content}" 
        for msg in messages
    ])
    
    prompt = ChatPromptTemplate.from_template("""
    你是公司的资深技术作家。
    请阅读下面的【对话历史】，特别是【tool】或【ai】提供的检索事实。
    
    【对话历史】:
    {history}
    
    基于上面的事实，写一篇结构清晰、通俗易懂的回答。
    """)
    
    chain = prompt | llm
    response = chain.invoke({"history": conversation_str})
    return {"messages": [response]}
```

#### 明确角色职责

**研究员（Researcher）**：
- 性格设定：严谨、只认数据
- 职责：分析用户意图，决定是否需要查知识库
- 装备：多种工具（知识库检索、通用问答等）

**作家（Writer）**：
- 性格设定：文采好、善于总结
- 职责：基于研究员提供的资料，撰写最终回答
- 无需工具，纯靠大模型能力

3. **工具调用顺序错误**：需要多工具协作时顺序混乱

## 6. 消息状态管理问题

### 问题描述

1. **消息覆盖而非累加**：新消息覆盖了旧消息
2. **状态结构不清晰**：不知道状态中应该包含哪些字段

### 解决方案

#### 使用 Annotated 实现消息累加

```python
from typing import Annotated, List, TypedDict
import operator
from langchain_core.messages import BaseMessage

class AgentState(TypedDict):
    """
    智能体之间传递的数据结构
    Annotated[List[BaseMessage], operator.add] 的意思是：
    当有新消息来的时候，不是覆盖旧消息，而是追加 (add) 到列表后面。
    """
    messages: Annotated[List[BaseMessage], operator.add]
    next_step: str  # 下一步指示
```

**关键点**：
- `Annotated[List[BaseMessage], operator.add]` 实现消息累加
- LangGraph 自动处理状态聚合

## 常见问题 FAQ

### Q1: 为什么检索不到内容？

**可能原因**：
1. 文档未正确入库 → 检查 `data/vector_db/` 目录
2. `project_id` 不匹配 → 确认当前选中的知识库
3. 向量数据库损坏 → 删除 `data/vector_db/` 重新入库

### Q2: 如何提高检索质量？

**优化方向**：
1. 调整 `CHUNK_SIZE` 和 `CHUNK_OVERLAP`
2. 换更好的 Embedding 模型（如 `text-embedding-3-large`）
3. 优化文档切分策略
4. 增加重排序（Rerank）步骤

### Q3: 如何支持更多文件类型？

在 `src/rag/etl.py` 的 `_select_loader` 方法中添加：

```python
elif suffix == ".xlsx":
    return YourExcelLoader(file_path)
```

### Q4: 多智能体为什么会循环调用？

**排查方法**：
1. 检查 `should_continue` 的逻辑
2. 确认 Researcher 的 System Prompt 是否正确引导
3. 考虑添加最大迭代次数限制

3. **性能瓶颈**：检索延迟过高（1.78s），主要在Embedding API调用

### 解决方案

#### 9.1 Precision计算修复

**问题代码**：
```python
# ❌ 错误：关键词数/文档数，可能>1
precision = len(found_keywords) / len(retrieved)
```

**修复后**：
```python
# ✅ 正确：相关文档数/检索文档数，范围0-1
relevant_docs = sum(1 for doc in retrieved if contains_keyword(doc, keywords))
precision = relevant_docs / len(retrieved)  # 范围: 0-1
```

**修复结果**：Precision从1.16降至0.787（正确范围）

#### P1优化详情：Embedding缓存

**修改文件**: `src/rag/retriever.py`

**核心改动**:
1. 新增 `CachedEmbeddings` 类包装OpenAIEmbeddings
2. 使用MD5哈希作为缓存键
3. LRU策略管理缓存大小（默认1000条）
4. 记录缓存命中率统计

**代码实现**:
```python
class CachedEmbeddings:
    """带缓存的Embedding包装器"""
    def __init__(self, embeddings, cache_size=1000):
        self.embeddings = embeddings
        self.cache: Dict[str, List[float]] = {}
        self.cache_size = cache_size
        self.cache_hits = 0
        self.cache_misses = 0
    
    def embed_query(self, text: str) -> List[float]:
        cache_key = hashlib.md5(text.encode('utf-8')).hexdigest()
        
        if cache_key in self.cache:
            self.cache_hits += 1
            return self.cache[cache_key]
        
        result = self.embeddings.embed_query(text)
        self.cache[cache_key] = result
        return result
```

## 10. 提示词工程与优化

### 问题描述

在多智能体系统中，提示词设计存在以下问题：

1. **提示词分散**：提示词散落在多个文件（nodes.py, generator.py等），难以统一管理和维护
2. **工具描述不够清晰**：LLM难以准确判断何时使用哪个工具
3. **缺少思考链引导**：没有引导LLM进行结构化思考
4. **多知识库场景支持不足**：提示词没有考虑多知识库隔离的场景
5. **Writer节点输出不规范**：回答结构不清晰，缺少来源引用机制

### 解决方案

#### 10.1 统一提示词管理模块

**新增文件**: `src/agent/prompts.py`

```python
class PromptManager:
    """提示词管理器"""
    
    # Researcher Agent 提示词

    # Writer Agent 提示词

**优势**：
- 集中管理所有提示词，便于版本控制
- 支持 A/B 测试不同的提示词版本
- 便于国际化和本地化
